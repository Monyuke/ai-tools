# agentic web検索

## プロセス

- 検索ワード考える
- 検索
- 検索結果のフィルタ（＝評価。駄目検索は結果が0になる）
  - 評価
    - 無関係
    - 補助情報
      - この補助情報があるからドキュメント化できる、はありえるのでは？って思ったけど実装が複雑になりそう。補助情報だけ積み上がっても無意味にコンテキストを圧迫するし、補助情報の品質で上位、とか無理だし利用は難しそう。
      - ただ、評価では出しておくと直接情報の判定がいい感じになりそう。
    - 直接情報
- 有効結果件数が10件いく、もしくは試行回数が限界に達したら終わる、でいいのでは。
- あと進捗はlogger関数受け取って処理がよい。stに依存しすぎてる。

検索ワードと検索を分離すると、既存語彙にしか対応できないのでは？
検索結果を受けて再検索ができなくなってしまう。
検索自体は人まとまりに行ったほうがいいのでは？
それか検索ワード考えるときに、サマリじゃなくて検索結果自体を与えるか。こいつの評価はいくつだって。
それかあれか、検索はagentでやらせて、その結果はサマリだけわたすか。それがいいかも。

## 出力

- 検索結果データ
  - 検索ワード
  - 有効結果件数
  - ワードの有効率
    - 1回最大10件返ってくるので、10件有効な結果があれば優秀な検索ワードということ。
